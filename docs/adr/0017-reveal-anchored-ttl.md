# 0017 - Reveal-anchored TTL (timer starts on first reveal)

**Status:** accepted

ADR-0002 burns a Poof at the earlier of its TTL or its reveal budget. ADR-0014 makes the TTL deadline visible to the recipient as a countdown and keeps it private by stamping an absolute `expiresAt` into the encrypted metadata at create time, deliberately rejecting both storing the raw duration and ever having the server publish remaining time. This ADR adds an opt-in mode in which the TTL clock starts at the first Reveal rather than at creation, for a true self-destruct: the link can wait unrevealed, then vanishes a set time after it is first read. It is in tension with ADR-0014 because a reveal-anchored Poof has no deadline at create time.

## Decisions

- **Opt-in, default off, per-Poof flag (`revealAnchored`).** The default stays creation-anchored (ADR-0002 unchanged). The web form exposes it in "more options" as the checkbox "start the timer on first reveal".
- **Budget stays authoritative and immediate.** The reveal-anchored TTL is a second "whichever comes first" burn trigger, exactly mirroring ADR-0002's "TTL or budget, whichever first," with the TTL clock rebased to the first Reveal. A budget-1 Poof therefore still burns on its single Reveal (the server-side TTL is moot there); the new server behavior bites only for budget >= 2 or unlimited, where later Reveals live on the time left from the first Reveal onward.
- **Unrevealed cap = the maximum TTL (30 days).** At create the DO arms its alarm at `created_at + 30d` so a never-revealed Poof still burns, honoring ADR-0002's "nothing is permanent." On the first successful Reveal the DO re-arms `expires_at = now + ttlMs` and reschedules the alarm.
- **The deadline stays private; only the duration is stored.** A reveal-anchored Poof writes `ttlMs` (duration) and `revealAnchored: true` into the encrypted metadata and omits `expiresAt`, because no deadline exists until the first Reveal. This is the case ADR-0014 explicitly rejected for creation-anchored Poofs; it is accepted here precisely because there is no absolute deadline to stamp at create.
- **The authoritative deadline is disclosed only in the successful Reveal response.** The Worker sets an `x-poof-expires-at` header only on the `ok` (content) branch of reveal, which is reached only past the turnstile and PIN gates. So for a PIN-protected Poof the deadline is disclosed strictly after a correct PIN produces a successful Reveal, to a caller who by then already holds the plaintext. Wrong-PIN, no-PIN, locked, turnstile-required, and gone responses carry no deadline. Arming likewise happens only on the first successful Reveal (the `reveals_used` 0 to 1 transition), so a wrong-PIN guess cannot start the clock.
- **The flag is encrypted-only; the cleartext wire format is unchanged.** Unlike `pinRequired` and `turnstileRequired`, the anchor mode does not change how a caller reveals (still a plain `POST /reveal`), only when the Poof dies, so a non-key-holder does not need it. Keeping it out of the cleartext envelope means zero wire-format change and maximum privacy.
- **Recipient countdown.** The web client uses `x-poof-expires-at` from the Reveal response to drive the post-reveal fuse, so it is exact for every revealer, including the 2nd-or-later viewer of a budget >= 2 Poof. Before revealing, the precard shows a static "timer starts when you reveal it (then Nm)" instead of a ticking fuse, since no deadline exists yet.

## Considered options

- **Let the TTL take over the burn for budget 1 (linger after read):** rejected. It keeps ciphertext at rest after the content has been read and muddies budget semantics; an immediate budget burn is better for at-rest privacy.
- **Compute the recipient fuse client-side as `now + ttlMs` with no server deadline:** rejected. Exact for budget 1, but a later viewer of a budget >= 2 Poof would overstate the time left by the gap since the first Reveal, up to a full TTL, leaving stale plaintext on an idle tab past the real deadline and defeating ADR-0014 auto-clear for that viewer. The single response header is cheap and makes auto-clear honest for everyone at near-zero privacy cost.
- **Publish the deadline in the cleartext envelope or non-consuming meta once armed:** rejected. It re-leaks the deadline to any link-holder polling meta, exactly what ADR-0014 chose to avoid. The reveal-response header limits disclosure to actual revealers.
- **A separate creator-set "reveal within" window instead of the fixed 30-day cap:** rejected for now, to avoid a second time control in the form. The fixed cap reuses the existing ceiling and can be revisited if requested.

## Consequences

- `ClipMeta` carries either `expiresAt` (creation-anchored, unchanged) or `ttlMs` plus `revealAnchored` (reveal-anchored), never both. Poofs created before this change are unaffected.
- The DO gains a `reveal_anchored` flag and stores `ttl_ms`, via additive `ADD COLUMN` migrations using the same idempotent guard as `require_turnstile` and `allow_viewer_delete`. `reveal()` arms on the first-reveal transition and returns the deadline on success.
- The Reveal response gains an `x-poof-expires-at` header on success only. `@p00f/core`'s `read()` surfaces it; the CLI can display it.
- The published wire format, discovery doc, and `llms.txt` are unchanged, because the flag is encrypted-only.
- The countdown for the 2nd-or-later viewer of a budget >= 2 reveal-anchored Poof is exact via the header. The only residual imperfection is if an intermediary strips the header, in which case the client falls back to `now + ttlMs` (loose by the join gap). Accepted and documented.
- The unrevealed cap (up to 30 days) and the post-reveal TTL (up to 30 days, since `MAX_TTL_MS` is 30 days) are two independent clocks, so a reveal-anchored Poof can live up to roughly 60 days in total in the pathological case of a long post-reveal TTL revealed near the end of the unrevealed cap. Accepted: capping total life would mean shrinking a creator's chosen post-reveal window based on how long the Poof sat unrevealed, which is more surprising than two clearly bounded clocks. The ~60-day corner requires deliberately choosing a long post-reveal TTL, an unusual choice for a "self-destruct shortly after reveal" feature.
- Storing `ttl_ms` server-side does not change the trust model (ADR-0001). The server already holds `created_at`, `expires_at`, the reveal budget and count, and the exact size in cleartext to enforce the lifecycle; for a creation-anchored Poof the server can already read the TTL as `expires_at - created_at`. `ttl_ms` is the same class of operational access-control data, stored explicitly only because a reveal-anchored Poof's deadline does not exist until the first Reveal. The zero-knowledge boundary (the content and the encrypted metadata blob, both unreadable because the Fragment Key never reaches the server) is unchanged. This mirrors ADR-0014, where `expiresAt` already lives both in the encrypted metadata (for the recipient) and as the DO's authoritative `expires_at` (for enforcement). A non-key-holder polling the public API actually learns less about a reveal-anchored Poof than a normal one, since the flag and duration are encrypted-only and reveal-response-only and no deadline exists pre-reveal.
